# Repository Guidelines

These instructions are for Codex and any Codex-started automation working in this repository.

## Language And Emoji Rules

All repository artifacts must be written in English. This includes source code, comments, docstrings, Markdown documentation, configuration values, prompt text, logs, errors, commit messages, and PR descriptions generated by Codex.

The only exception is conversational chat between the user and Codex. Codex may reply in the user's language in chat, but generated or edited files must remain English.

Do not use emojis in source code, comments, string literals, logs, errors, configuration files, prompt templates, commit messages, or PR descriptions. Emojis are allowed only in Markdown documentation, README files, and files under `docs/`; avoid adding them unless the user explicitly asks.

## Supabase CLI And Migrations

Non-negotiable: Codex must never execute, schedule, delegate, or indirectly trigger any terminal command that runs the Supabase CLI or Supabase-managed database migrations on the user's machine.

Forbidden examples include `npx supabase`, `pnpm dlx supabase`, `pnpm exec supabase`, `yarn supabase`, `bun supabase`, a `supabase` binary on `PATH`, and any `package.json`, shell, Make, Just, Docker, or workspace command whose effect is to run Supabase CLI or migration tooling.

Codex must not run migration flows such as `migration new`, `migration list`, `db push`, `db pull`, `db reset`, repair, squash, or diff. Codex must not run Supabase flows such as `link`, `start`, `stop`, `status`, `gen types`, secrets, or seeding.

Codex must not create, rename, copy, or replace migration files under `supabase/migrations/` as a substitute for the user's CLI-generated migration stub. Only the user may create new migration files there by running `npx supabase migration new <migration_name>` or an equivalent user-run Supabase CLI flow.

When a Supabase command is needed, Codex may only print the exact command for the user to run, then halt dependent work until the user explicitly confirms the result. After the user confirms the CLI created a migration file and provides or confirms the filename, Codex may edit that existing migration file.

If there is any doubt that a command might invoke Supabase or touch migrations, do not run it. Print the command for the user instead.

## Application Architecture

This project uses Next.js App Router with feature-based colocation under `src/app`. Route segments own their UI, hooks, and helpers next to `page.tsx` and `layout.tsx`; shared code lives in top-level `src/` folders.

Expected layout:

```text
src/app/
  layout.tsx
  globals.css
  (pages)/
    (landing)/
      page.tsx
      _components/
      _hooks/
      _lib/
    (dashboard)/
      layout.tsx
      _components/
      dashboard/
        page.tsx
      clients/
        page.tsx
        [id]/
          page.tsx
          _components/
          _hooks/
          _lib/
    auth/
      _components/
  api/

src/components/
src/hooks/
src/lib/
src/services/
src/utils/
src/types/
```

Use route groups in parentheses, such as `(pages)` or `(dashboard)`, for organization only; they must not imply URL path segments. Use folders without parentheses for real URL segments. Use `_components/`, `_hooks/`, and `_lib/` for private route-local colocation.

Keep `page.tsx` thin: compose feature UI from colocated modules and keep domain logic, side effects, and integration boundaries in hooks, server modules, `src/services/`, or `src/lib/`. Promote route-local modules into shared `src/` folders only after a second feature needs them.

Use the `@/*` alias for imports from `./src/*` when it improves clarity.

## State Management

Use TanStack Query for server state: fetching, caching, synchronization, mutations, invalidation, and remote data where the server is the source of truth.

Use Zustand for client-only UI and workflow state: modals, wizard steps, local selections, layout state, and other ephemeral state that should not live in the remote data cache.

Do not use Zustand as a substitute for TanStack Query for server-backed data. Do not put purely local UI state into TanStack Query when a small local store is clearer.

## Codex Skill Usage

Project skills live under `.agents/skills/<skill-name>/SKILL.md`. When a trigger below applies, Codex must read the relevant `SKILL.md` before editing files. If multiple triggers apply, read the narrowest relevant skills first and use each skill only for its own responsibility.

Available project skills:
- `ui-ux-pro-max`: UI/UX design intelligence for web and mobile, including styles, color palettes, font pairings, product types, UX guidelines, charts, and patterns across React, Next.js, Tailwind, shadcn/ui, and HTML/CSS.
- `tailwind-best-practices`: Tailwind utility implementation, responsive classes, dark mode, and Tailwind config.
- `shadcn-ui`: shadcn/ui component selection, installation, customization, variants, and accessibility.
- `tanstack-query`: server state, query keys, mutations, invalidation, retries, optimistic updates, and async cache patterns.
- `zustand`: typed client stores, selectors, middleware, persistence, and local workflow state.
- `react-best-practices`: React and Next.js rendering, performance, hydration, Suspense, memoization, and component refactors.
- `designing-tests`: unit, integration, and E2E test strategy, setup, fixtures, mocks, and reliability.
- `designing-apis`: REST and GraphQL API contracts, route handlers, status codes, errors, versioning, and documentation.
- `error-handling`: application errors, structured logging, retries, circuit breakers, observability, and graceful degradation.
- `karpathy-guidelines`: baseline coding discipline for surgical changes, assumptions, simplicity, and verifiable success criteria.
- `nextjs-docs`: project documentation, README-style output, architecture docs, setup guides, and deployment docs.
- `prompt-master`: prompts for LLM, coding, image, video, or other AI tools.

## Mandatory Skill Triggers

Before the first edit in a code-writing task, scan the request and target files for these triggers:

- Use `ui-ux-pro-max` when planning, building, designing, implementing, reviewing, fixing, improving, optimizing, enhancing, refactoring, or checking UI/UX for websites, landing pages, dashboards, admin panels, e-commerce, SaaS, portfolios, blogs, mobile apps, components, forms, tables, charts, navigation, accessibility, animation, responsive behavior, typography, color, spacing, interaction states, shadows, or gradients.
- Use `tailwind-best-practices` when editing `className`, `tailwind.config.*`, `globals.css`, `@theme` blocks, utility strings, or `cn()` composition.
- Use `shadcn-ui` when importing from `@/components/ui/*`, adding shadcn components, or changing shadcn variants and behavior.
- Use `tanstack-query` when importing `@tanstack/react-query`, changing `useQuery`, `useMutation`, `useInfiniteQuery`, query keys, cache invalidation, prefetching, retries, or optimistic updates.
- Use `zustand` when importing `zustand`, creating or modifying a store, or implementing persisted/global client UI state.
- Use `react-best-practices` when refactoring component performance, rerender behavior, `useMemo`, `useCallback`, `React.memo`, Suspense, hydration, or data waterfalls.
- Use `designing-tests` when editing `*.test.*`, `*.spec.*`, `__tests__/`, test setup, fixtures, mocks, coverage, or reliability.
- Use `designing-apis` when creating or modifying `src/app/api/**`, request/response shapes, status codes, REST, GraphQL, or API versioning.
- Use `error-handling` when adding `try`/`catch`, error boundaries, retries, logging, instrumentation, or external-call failure modes.
- Use `nextjs-docs` when generating or updating project docs, README files, architecture docs, setup guides, or deployment guides.
- Use `prompt-master` when writing, improving, or adapting prompts for AI tools.
- Use `karpathy-guidelines` for code changes not covered by a narrower trigger, large multi-file changes, refactors, or unclear scope.

Common stacks are allowed when responsibilities differ: UI work usually needs `ui-ux-pro-max` for UX direction, `design-system` for tokens and values, and `tailwind-best-practices` for implementation; shadcn UI work may also need `shadcn-ui`; data-fetching features often need `tanstack-query` plus `error-handling`; new route handlers usually need `designing-apis` plus `error-handling`.

## Skill Boundaries

Let each skill decide only its own concern:

- `design-system` decides what design tokens and visual values to use.
- `ui-ux-pro-max` decides product-level UI/UX direction, page and component patterns, style selection, palette and font recommendations, accessibility, responsive behavior, interaction quality, animation, chart choices, and visual polish.
- `tailwind-best-practices` decides how to implement styling with Tailwind.
- `shadcn-ui` decides component-library usage.
- `tanstack-query` decides server-backed async state patterns.
- `zustand` decides client-only store patterns.
- `react-best-practices` decides React rendering and performance structure.
- `designing-tests` decides testing structure and quality.
- `designing-apis` decides public API shape and contracts.
- `error-handling` decides operational failure behavior.
- `nextjs-docs` decides documentation structure.
- `prompt-master` decides prompt quality.
- `karpathy-guidelines` provides general implementation discipline and does not replace narrower skills.

If skills conflict, follow the most specific skill for the decision at hand. For design versus implementation conflicts, `design-system` decides the token or value, and implementation skills adapt to it.

## Subagents

Use specialized subagents only when the current Codex environment exposes an appropriate tool. Prefer one primary subagent per task. Use a Next.js-focused subagent for App Router structure, route groups, Server/Client Component boundaries, caching, `layout.tsx`, or `page.tsx` architecture. Use a TypeScript-focused subagent for advanced generics, conditional types, mapped types, or end-to-end type safety.

Do not delegate work to bypass repository rules. Any subagent, background command, or automation started by Codex must obey this file, especially the Supabase prohibition.
